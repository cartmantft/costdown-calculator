# 연구 노트 (Phase 0)

## 결정 및 근거

### 삭제 영속성: 로컬 스토리지 일관성 유지
- **Decision**: 계산 히스토리는 로컬 스토리지 단일 소스(현재 limit 10)로 유지하고, 삭제 시 즉시 동일 스토리지에서 항목을 제거한 후 메모리 상태와 동기화한다.
- **Rationale**: 단일 저장소를 사용하면 재실행 후 부활 문제를 방지하고, 리스트 순서 보존을 보장한다.
- **Alternatives considered**:  
  - 별도 캐시/메모리 선행 → 앱 재실행 시 불일치 가능성 높음.  
  - 세션 스토리지 사용 → 앱 종료 후 삭제 상태가 사라져 요구사항 불충족.

### 줌 차단: 뷰포트 스케일 고정
- **Decision**: 뷰포트에 `initial-scale=1, maximum-scale=1, user-scalable=no`를 적용하고 멀티터치 줌 제스처를 전역에서 비활성화한다.
- **Rationale**: 출시 가이드에서 요구하며, 레이아웃 붕괴를 막기 위한 표준 접근 방식이다.
- **Alternatives considered**:  
  - 화면별 조건부 차단 → 누락 위험이 있어 검수 실패 가능.  
  - JS 이벤트로 제스처 막기만 적용 → 메타 뷰포트 미설정 시 일부 브라우저에서 확대 가능.

### 브랜드 노출: 브릿지/내비 아이콘 일관성
- **Decision**: 브릿지 뷰와 공통 내비게이션 바 모두에서 동일한 앱 아이콘과 이름, 브랜드 색을 로드하고, 로드 실패 시 이름 기반 대체 표시와 재시도 로직을 둔다.
- **Rationale**: 출시 필수 조건이며, 로드 실패 시에도 진입을 막지 않아야 한다.
- **Alternatives considered**:  
  - 브릿지 뷰만 브랜드 표시 → 내비게이션 불일치로 거부 가능.  
  - 로드 실패 시 진입 차단 → 사용자 이탈 및 검수 지연 초래.
